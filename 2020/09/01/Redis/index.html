**NoSql**:不依赖业务逻辑方式进行存储,只是通过简单的key-value模式存储，因此大大的增加了数据库的扩展能力

​ 特点：不遵循SQL标准（select等）、不支持ACID（原子性、一致性等）、远超SQL的性能

Redis

​ 与 memcached 的区别Redis会周期性的把更新的数据写入磁盘或者把修改操作写入追加的记录文件，并且在此基础上实现了master-slave(主从)同步。

​ 类型：string(字符串)、list（链表）、set（集合）、zset（有序集合）、hash（哈希类型）

​ 多路id复用

​

Redis事务

​ 事务中的所有命令都会序列化、按顺序地执行。事务在执行的过程中，不会被其他客户端发送来的命令请求所打断。

​ 主要作用：是为了串联多个命令防止别的命令插队

Redis持久化

​ 将数据保存到磁盘中,redis是做缓存来使用的，是一个非关系型数据库，将数据存储在内存中

RDB

​ 在指定的时间间隔内将内存中的数据集快照写入磁盘，恢复时将快照文件直接读到内存中

​ 备份如何执行的

​ Redis会单独创建（fork）一个子进程来进行持久化，会先将数据写入到一个临时文件中，待持久化过程都结束了，再用这个临时文件替换上次持久化好的文件。整个过程中，主进程是不进行任何IO操作的，这就确保了极高的性能如果需要进行大规模数据的恢复，且对于数据恢复的完整性不是非常敏感，那RDB方式要比AOF方式更加的高效。RDB的缺点是最后一次持久化后的数据可能丢失。

​ 在redis中保存的文件名称为:dump.db

​ 保存方式:

​ 1.通过使用保存策略

​

​ 在固定的时间之内修改相应的参数，则启用保存策略进行保存

​ 缺点：如果没有在规定的时间修改相应的次数，则不会进行保存

AOF

​ 以日志的形式来记录每个写操作，将Redis执行过的所有写指令记录下来(读操作不记录)，只许追加文件但不可以改写文件，Redis启动之初会读取该文件重新构建数据，换言之，Redis重启的话就根据日志文件的内容将写指令从前到后执行一次以完成数据的恢复工作。

​ RDB和AOF同时开启，则redis听AOF

​ 优点：备份机制更稳健，丢失数据概率低

​ 缺点：

​ 和RDB相比占用更多磁盘空间；

​ 恢复数据比较慢

RDB和AOF的区别:

​ RDB保存的是数据，AOF保存的是指令，相对来说RDB节省一些磁盘

薪火相传

​ 一台从服务器，可以设置成为另一台的主服务器：

​ slaveof no one（命令含义：将这台从服务器设置为不做任何服务器的从机）

​ 反客为主：当一个主服务器宕机后，后面的从服务器可以立刻升为主服务器，其后面的从服务器不用做任何修改，将会清除之前的数据

哨兵模式

​ 反客为主的自动版：在后台监听主机是否故障，如果故障则投票决定将哪一台从服务器转换为主服务器
